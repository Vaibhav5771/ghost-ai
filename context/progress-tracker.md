## Session Notes (18)
- The Templates modal lives inside `<CanvasFlow>` (below `RoomProvider` and `ReactFlowProvider`), not in `<WorkspaceShell>` alongside the navbar button. Reason: the import path needs `useLiveblocksFlow`'s `onNodesChange`/`onEdgesChange`/`onDelete` and `useReactFlow`'s `fitView`, all of which require the Liveblocks + React Flow provider trees. Moving the open state into `WorkspaceShell` and passing `templatesOpen`/`onTemplatesOpenChange` down through `CanvasWrapper` → `CanvasFlow` keeps the navbar button outside the providers (where it belongs) while letting the modal render inside them.
- Cleared the canvas via `onDelete({ nodes, edges })` rather than emitting per-item `{ type: "remove", id }` changes. `onDelete` is the same API user-initiated deletes flow through — it drives the collaborative Storage sync path with one round-trip instead of N — and matches the existing pattern the codebase already uses for the delete-key path.
- Added template nodes through `onNodesChange([{ type: "add", item }, ...])` instead of writing to `storage.flow.nodes` directly, mirroring the drop handler (Feature 12 note): a direct `set(id, plainObject)` bypasses `@liveblocks/react-flow`'s `toLiveblocksInternalNode` wrapper, so the stored value would lack the `setLocal`/atomic-position machinery and crash on the first drag/measure. Going through `onNodesChange` gives us the LiveObject wrap + `NODE_BASE_CONFIG` for free.
- Cloned each template's `nodes`/`edges` (`{ ...node, position: { ...node.position }, data: { ...node.data } }`) before dispatching. `CANVAS_TEMPLATES` is a module-level constant; if imports handed those objects straight to Liveblocks, later drag/label mutations on one imported canvas would scribble on the shared constant and pollute every subsequent import in the same session.
- `fitView` runs inside `requestAnimationFrame` because React Flow measures node bounds via `ResizeObserver` on the DOM elements the reconciliation just produced — calling `fitView` in the same tick uses stale (or zero) bounds and lands the camera at a nonsense zoom. One rAF is enough on this pipeline (Liveblocks-react-flow flushes the add batch synchronously through React Flow's state), and the 400ms duration matches the ergonomic feel of the existing zoom controls (Feature 17).
- Preview intentionally skips React Flow (spec: "no React Flow instance needed") — pure SVG with `preserveAspectRatio="xMidYMid meet"` keeps all three template cards rendering identically regardless of the card's actual CSS width, and dodges the ~200KB React Flow bundle per preview instance. The layout math is straightforward: bounds → uniform scale to fit `(PREVIEW_WIDTH - 2*padding) × (PREVIEW_HEIGHT - 2*padding)` → center the scaled bounds inside the viewport.
- Preview edges are drawn between projected node **centers** rather than edge endpoints. Using `sourceHandle`/`targetHandle` would require replicating React Flow's handle-position math (source-right vs. target-left) for a preview that already loses visual fidelity on scale — center-to-center is simpler and reads correctly at the preview's zoom level.
- Both `sourceHandle: "right"` and `targetHandle: "left"` on template edges match the four-handle-per-node source setup from Feature 16. Since all four handles are `type="source"` under `ConnectionMode.Loose`, "target" is really just another source ID — pinning `right → left` gives a predictable left-to-right flow on the pre-composed template layouts without hunting for the closest handle at render time.

The shape panel is bottom-center (`left-1/2 -translate-x-1/2`), so they never share horizontal space — the spec's "sit above the shape panel" reads as a z-order requirement, satisfied by both bars being `z-10` siblings inside the canvas root (later sibling paints on top). Trying to stack them vertically at bottom-center would collide with the minimap and force the shape panel to move, which the scope limits explicitly forbid.

- `useKeyboardShortcuts` guards against firing when a `mod` (Cmd/Ctrl) key is held for `+`/`=`/`-` — otherwise `Ctrl+=` (browser zoom on some keymaps) or `Ctrl+-` would fire canvas zoom too. `Cmd/Ctrl+Z`/`Y` handlers `preventDefault` so the browser doesn't also run its native form undo when the canvas is focused.
- The editable-field guard uses `HTMLElement.isContentEditable` rather than the `contenteditable` attribute string — attribute-based checks miss inherited `contenteditable` from ancestors (React Flow's node containers) and treat `contenteditable="false"` as truthy. `isContentEditable` is the resolved boolean on the element instance.
- Undo/redo hooks are called at the `CanvasFlow` level (not inside `CanvasControls`) so the same `undo`/`redo` references thread through both the control-bar buttons and the keyboard-shortcut hook — one source of truth, and it lives where `useReactFlow()` also lives. `CanvasControls` still calls `useUndo`/`useRedo`/`useCanUndo`/`useCanRedo` because the button state depends on `canUndo`/`canRedo` which we don't want to plumb through props for a co-located component; the hook returns the same underlying room-history reference either way.
- No manual `stopPropagation` on the control-bar buttons (unlike node-color toolbar) — they sit on the pane background outside any node's `nodrag/nopan` scope, and the pane doesn't start a selection-box drag on a button `mousedown` because React Flow only pane-drags when the target is the pane itself.

## Session Notes (16)
- `useUpdateNodeInternals(id)` on mount was necessary because React Flow only recomputes `internals.handleBounds` when a node's DOM dimensions change (via its ResizeObserver). Switching from 2 unnamed handles to 4 labeled handles in source code doesn't resize the node, so pre-existing nodes in the Liveblocks room kept stale bounds without an "left"/"right" entry — new edges then failed the `getHandle$1` lookup and logged React Flow error 008. Firing `updateNodeInternals` once per node on mount forces a remeasure.
- All four handles use `type="source"` (not one target + three sources) because `ConnectionMode.Loose` is already set in `canvas-flow.tsx` — Loose lets a source drop onto another source. Splitting types would leak asymmetry into the UI (a "target-only" side users can't drag from) and require extra logic to hide it.
- Handles stay `pointer-events: auto` at all times; only `opacity` fades on hover. Setting `pointer-events: none` when hidden broke connection drag: once the cursor left the node during a wire drag, React Flow's `pointerdown` handler on the (now non-interactive) handle stopped receiving events and the wire snapped off. Fading only opacity keeps the drag lifecycle intact — an invisible handle catching a click is the acceptable trade-off.
- Arrowhead marker is rendered inline **inside each edge's SVG** (unique id per edge, `orient="auto-start-reverse"`, `markerUnits="strokeWidth"`) rather than as a shared `<defs>` in the root. This lets the marker fill follow the current stroke color (rest vs. active) without SVG's `context-stroke` paint value, which has spotty browser support. The trade-off is one marker element per edge, which is fine at canvas scale.
- Hit-area trick: two `<path>` elements sharing the same `d`. Underneath, a `strokeWidth=22` fully-transparent path catches pointer events (`onMouseEnter`/`onDoubleClick`); on top, a `strokeWidth=1.5` visible path with `pointer-events: none` so it never blocks the hit path. Same pattern React Flow's `BaseEdge` uses with `interactionWidth`.
- Path midpoint comes straight from `getSmoothStepPath` (returns `[path, labelX, labelY, offsetX, offsetY]`). Feature spec called this out explicitly: **don't** compute midpoint manually from `(sourceX + targetX) / 2` — smooth-step routing has kinks and a straight average lands off-path when source/target are on the same axis.
- Editing input auto-grows via a hidden mirror `<span>` (`whiteSpace: "pre"`, same font size/family) whose `offsetWidth` is read after each `draft` change. The `<input>`'s `size` attribute uses `ch` units and drifts across fonts; measuring the mirror is font-accurate. Effect runs only when `editing` is true so the mirror doesn't stay mounted.
- `updateEdgeData(id, { label })` is the equivalent of the node color toolbar's `updateNodeData` — React Flow internally converts it into an `onEdgesChange` "replace" event that `@liveblocks/react-flow` reconciles as one atomic Storage update. Direct LiveObject writes on the edge would work too but would bypass the type-safe API and duplicate the pattern used everywhere else in the canvas.
- `setState` inside the "enter editing" effect tripped `react-hooks/set-state-in-effect`; moved the `setDraft(label)` into a `startEditing` callback fired from `onDoubleClick`. The remaining effect only calls DOM methods (`focus`/`select`), not `setState`.

- Kept `color` (background) and `textColor` as two independent optional fields on `CanvasNodeData` rather than storing a `paletteId` — the render path (`ShapeVisual`, label span) already treats `color` as a raw CSS color string, and writing both fields keeps the node self-describing without a palette lookup at render time. `activeBackground={data.color}` compares strings so the active-swatch outline still works.
- Swatch writes go through the same LiveObject path label edits use (`storage.get("flow").get("nodes").get(id).get("data").set(...)`) so we stay inside the collaborative canvas state and skip `onNodesChange`, which would replace the whole node and re-create the LiveObject (same reasoning as Feature 14).
- Toolbar interaction guards: `nodrag nopan` classes + `stopPropagation` on `mousedown`/`pointerdown`/`wheel`/`dblclick`. `nodrag` alone was not enough during Feature 14 either — React Flow 12 sometimes claims the pointer for selection-box drag before the class-based short-circuit fires.
- Hover glow uses local `useState` for `hoveredId` rather than CSS `:hover` because inline styles can't express pseudo-selectors and adding a global class per palette entry would spread the palette across two files. `box-shadow: 0 0 4px 1px <textColor>` keeps the glow tight (1px spread, 4px blur) — the spec called out "tight and controlled, not overly blurred".
- Placeholder color: switched from a hardcoded `rgba(255,255,255,0.35)` to `color: textColor; opacity: 0.45`, so light-text-on-dark palettes look right AND dark-text-on-light palettes (amber, cyan) still get a legible dimmed placeholder. Using `opacity` on the span is fine here because the span has no other children.
- Toolbar placement uses `bottom: calc(100% + 10px)` on the node's `position: relative` root — same coordinate frame the shape and handles use — so it tracks the node during drag and never overlaps the shape. `zIndex: 10` keeps it above the shape/handles without interfering with the node-resizer handles (which are outside this DOM subtree).

## Session Notes (14)
- `<NodeResizer>` is added inside the custom node and only sets `isVisible`/`min*` props; it dispatches `dimensions` + `position` changes through the same `onNodesChange` that `useLiveblocksFlow` already wraps, so resize syncs to Storage without any custom mutation. Verified the `NODE_BASE_CONFIG` in `@liveblocks/react-flow` makes `position` atomic and treats `width`/`height` as deep-synced fields.
- Label writes can't go through `onNodesChange` with a `replace` change because `useLiveblocksFlow` syncs `data` as a (deep) LiveObject — replacing the whole node would re-create the LiveObject and break references for in-flight selection/drag state. Direct LiveObject write via `useMutation` (`storage.get("flow").get("nodes").get(id).get("data").set("label", value)`) is the path that keeps the existing node identity and patches just the label key.
- Existing `data.label` reads come from the controlled `nodes` array returned by `useLiveblocksFlow`, which already subscribes to Storage. Using it as the textarea's `value` keeps the typing experience real-time without local mirror state and lets concurrent updates from other clients show up live.
- Label + textarea share one `position: absolute; inset: 0; display: flex; align-items: center; justify-content: center` container; swapping the inner child in place is what satisfies "no layout shift" — sizes/positions are identical between the two modes.
- Textarea uses `rows={1}` and auto-resizes on input by setting `style.height = "auto"` then `scrollHeight + "px"`. Initial mount runs the same measurement inside the `editing` effect so labels that were already multi-line open at the correct height.
- `className="nodrag nopan nowheel"` plus `stopPropagation` on `mousedown`/`pointerdown`/`wheel` was needed (just `nodrag`/`nopan` was not enough in 12.10.2 — wheel scrolling inside the textarea still bubbled to the pane zoom, and React Flow occasionally claimed the pointer for selection-box drag before nodrag could short-circuit).
- Initial `useEffect` had a second effect that called `setEditing(false)` when the node became deselected. React 19's `react-hooks/set-state-in-effect` flagged it, and it was redundant anyway — clicking off the node loses focus on the textarea, which triggers `onBlur` → `setEditing(false)`. Removed.
- Empty-label rendering: when `data.label` is falsy the span renders `"Add label"` at the same centered position with a dimmer color (`rgba(255,255,255,0.35)`), so the placeholder occupies the same slot a real label would, matching the spec's "same centered position" requirement. The textarea also carries `placeholder="Add label"` so the same hint shows while editing an empty node.

## Session Notes (13)
- Shape rendering split into a shared `ShapeVisual` component so the same renderer powers both the node and the drag preview — keeps shapes visually identical and avoids drift.
- `preserveAspectRatio="none"` + `vectorEffect="non-scaling-stroke"` is what lets diamond/hexagon/cylinder stretch to arbitrary node sizes without distorting the stroke width.
- HTML5 drag image element needs to be in the DOM and rendered (not `display: none`) when `setDragImage` is called, so the off-screen previews use `position: absolute; left: -10000px` rather than `display: none` or `visibility: hidden`.
- Drag image is centered on the cursor by passing `(width/2, height/2)` as the offset to `setDragImage` — matches the drop handler which centers the node on `clientX`/`clientY`.
- Fix carried over from feature 12: `canvas-flow.tsx` previously dropped nodes via a custom `useMutation` that did `storage.get("flow").get("nodes").set(id, node as any)`. That stored a plain object, so as soon as React Flow tried to drag/measure/select the node `@liveblocks/react-flow` crashed with `node.setLocal is not a function` or `node.get is not a function`. Insertion now goes through `onNodesChange([{ type: "add", item: node }])`, which is the library's own path and runs the internal `toLiveblocksInternalNode` wrapper (LiveObject + `NODE_BASE_CONFIG` with atomic `position` and local-only `selected`/`dragging`/`measured`/`resizing`).
- Legacy data heal: rooms persisted by the broken handler still contain plain-object nodes. `canvas-flow.tsx` adds a `removeLegacyNodes` mutation + `useEffect` that runs once on mount — it scans `storage.flow.nodes` for entries missing a LiveObject `.get` method, deletes them, then re-adds them through `onNodesChange` so they get re-wrapped. Subsequent loads are no-ops. The mutation runs synchronously inside the effect, before React Flow's ResizeObserver fires, so the first measurement pass operates on healthy LiveObjects.
- MiniMap shapes: by default `<MiniMap>` renders every node as a rounded rect. `components/editor/canvas/minimap-shape.tsx` is a custom `nodeComponent` that subscribes to each node's data via `useNodesData(id)` and emits the matching SVG primitive: `<rect rx={5}>` for rectangle, `<rect rx={height/2}>` for pill, `<ellipse>` for circle, `<polygon>` for diamond/hexagon, and a body `<path>` + top `<ellipse>` for cylinder. Passed to `<MiniMap nodeComponent={MinimapShape}>` so the minimap visualization matches the canvas.

## Architecture Decisions

- Using shadcn/ui on Tailwind v4: CSS variables defined in globals.css via `@theme inline` (no tailwind.config.js)
- Dark-only theme: CSS variables set on `:root`, no light-mode override
- shadcn components live in `components/ui/` — generated by CLI, not modified after installation
- cn() helper in `lib/utils.ts` uses clsx + tailwind-merge
- Animation utilities (animate-in, fade-in-0, zoom-in-95, slide-in-from-*) defined via `@utility` in globals.css for Tailwind v4 compatibility
- `components.json` present at project root for shadcn CLI configuration
- Editor chrome components live in `components/editor/` — hand-authored, not generated
- Project sidebar is `fixed` positioned (floats above canvas, does not push content); slide-in via CSS transform transition
- Auth uses Clerk v7 (`@clerk/nextjs` ^7.3.0); proxy.ts (not middleware.ts) per Next.js 16 convention
- ClerkProvider is inside `<body>` per Clerk v7 requirement; uses `dark` theme + CSS var overrides
- Clerk appearance uses `shadcn` + `dark` themes from `@clerk/ui/themes`, with `@clerk/ui/themes/shadcn.css` imported in globals.css
- Auth pages use route group `app/(auth)/` with shared two-panel layout; catch-all segments `[[...sign-in]]` and `[[...sign-up]]` for Clerk's multi-step flows
- Protected-first middleware: all routes protected except NEXT_PUBLIC_CLERK_SIGN_IN_URL, NEXT_PUBLIC_CLERK_SIGN_UP_URL, and `/api/projects(.*)`. Project API routes enforce Clerk auth inside route handlers so they can return the Feature 06-required JSON `401`.
- Project metadata is stored in PostgreSQL through Prisma. Canvas JSON remains externalized for future blob storage via `Project.canvasJsonPath`.
- Prisma Client singleton lives in `lib/prisma.ts`, uses `accelerateUrl` for `prisma+postgres://` URLs and `@prisma/adapter-pg` for direct Postgres URLs, and is cached on `globalThis` outside production.
- Editor workspace URLs use `/editor/[projectId]`; Feature 07 aligns the project ID and Liveblocks room ID by allowing `POST /api/projects` to accept a sanitized slug-style client-generated ID while preserving Prisma's default ID strategy when no ID is supplied.
- `lib/project-access.ts` is the server-side access layer; `lib/project-data.ts` is the server-side data-fetch layer for the sidebar. The two are intentionally separate so the workspace page can do one access check and one full project-list fetch in parallel.
