## Session Notes (26 — integration debugging, post-implementation)

**Prisma Accelerate + migrations**
- `DATABASE_URL` in this project is a Prisma Accelerate URL (`prisma+postgres://accelerate.prisma-data.net/?api_key=...`). Accelerate is an HTTP proxy — it supports DML (`SELECT`/`INSERT`/`UPDATE`/`DELETE`) but the underlying role has **no DDL privileges**. `CREATE TABLE`, `CREATE INDEX`, etc. fail with `42501 permission denied for schema public`. This applies to `prisma.$executeRawUnsafe` through the runtime client too — not just the migrate CLI.
- **Migrations require the direct URL.** Set `DIRECT_URL=postgres://TENANT_ID:sk_.....@pooled.db.prisma.io:5432/postgres?sslmode=require` (get from console.prisma.io → Connect tab). `prisma.config.ts` now falls back: `url: process.env.DIRECT_URL ?? process.env.DATABASE_URL`. The runtime Prisma client in `lib/prisma.ts` still uses the Accelerate `DATABASE_URL` — only migrations use the direct URL.
- `prisma.config.ts` uses `dotenv.config({ path: ".env.local" })` explicitly. Default `import "dotenv/config"` loads `.env` only, which doesn't exist in this project. The old config file silently returned `undefined` for `DATABASE_URL` and produced misleading "datasource.url property is required" errors.
- The Prisma Console's SQL Editor **also uses the Accelerate-restricted role** — pasting `CREATE TABLE` in it fails silently (the parts that don't need DDL, like INSERT into `_prisma_migrations`, succeed; the CREATE TABLE errors show as a red dot at the failing line). Do not use it for DDL. Fixed a run of the `20260714000000_add_task_run` migration where the migration row was inserted but the table was never created — cleaned up with `scripts/fix-migration-state.ts` (`DELETE FROM "_prisma_migrations" WHERE migration_name = '...'`).

**Prisma Client construction**
- The generated client in `app/generated/prisma/client` accepts `accelerateUrl` (not `datasourceUrl`) as the constructor option. `lib/prisma.ts` uses `new PrismaClient({ accelerateUrl }).$extends(withAccelerate())`. Ad-hoc scripts must use the same option name or Prisma throws `PrismaClientConstructorValidationError: Unknown property datasourceUrl`.

**AI SDK + Groq structured outputs**
- Route: `useRealtimeRun` from `@trigger.dev/react-hooks` uses SSE with a scoped public token. The `run` object updates in real time; `run.status` transitions through `EXECUTING` → `COMPLETED`/`FAILED`/`CRASHED`/etc.
- **`onComplete` callback is unreliable in v4.5.4.** It didn't fire in dev even for successful runs. Use a `useEffect` on `run?.status` instead — matches a `terminalStatuses` list and does the cleanup + push final chat message. This is the pattern in `workspace-shell.tsx` now.
- `generateObject` from the `ai` package (`ai@^6`) removed the old `mode: "json"` param. It is not a valid option and TypeScript rejects it. Only `providerOptions.groq.structuredOutputs: boolean` controls schema-vs-object mode via the Groq SDK.
- **Groq model support for structured outputs is model-specific**, not provider-wide. `llama-3.3-70b-versatile` does NOT support `response_format: json_schema` — the API returns `This model does not support response format 'json_schema'`. Falling back to `structuredOutputs: false` gets `json_object` mode, but that in turn (a) requires the word "json" to appear in the messages and (b) does not enforce the schema, so `generateObject` throws `NoObjectGeneratedError` when the model's ad-hoc JSON doesn't match Zod exactly.
- **Groq models that DO support `json_schema`** (as of this session): `openai/gpt-oss-20b`, `openai/gpt-oss-120b`, `moonshotai/kimi-k2-instruct`, `meta-llama/llama-4-maverick-17b-128e-instruct`, `meta-llama/llama-4-scout-17b-16e-instruct`. `trigger/design-agent.ts` now uses `meta-llama/llama-4-scout-17b-16e-instruct` — fastest of the schema-supporting models on the free tier.
- **Groq strict JSON schema requires all properties in `required`.** Optional fields must use `z.string().nullable()` (JSON schema: `"type": ["string", "null"]`) rather than `.optional()` (omit from `required`). Groq error: `properties/edges/items/required: 'required' is required to be supplied and to be an array including every key in properties`. Updated `geminiSchema` accordingly; edge `label` conversion at the broadcast site does `label: edge.label ?? undefined` because `CanvasAction.label` is `string | undefined`.
- Removed `paletteEnum.default("default")` on the node schema at the same time — Zod defaults don't survive the strict-mode JSON schema conversion cleanly and are unnecessary given the palette lookup already falls back to `DEFAULT_NODE_BACKGROUND`/`DEFAULT_NODE_TEXT`.

**Alternative providers considered**
- Google Gemini (`@ai-sdk/google` already installed, `GEMINI_API_KEY`) — best schema support but the free-tier rate limits are too tight for iterative dev (15 RPM, 1500/day). Was originally used in Feature 23, switched away for exactly this reason.
- OpenRouter (`@openrouter/ai-sdk-provider`) — free-tier models with `:free` suffix, aggregator; not tried but noted as viable fallback if Groq's schema-supporting models become insufficient.
- Cerebras (`@ai-sdk/cerebras`) — free tier, fastest inference; noted as viable fallback.

## Session Notes (23)
- `liveblocks.setPresence` is called best-effort (try/catch) because it only affects users who have an active WebSocket connection to the room. The AI task is a background process with no WebSocket connection, so the call typically silences. The presence effect on the canvas comes from the `ai:thinking` broadcast event instead — the client renders its own "Ghost AI is designing…" overlay driven by that event, independent of Liveblocks presence state.
- `broadcastEvent` is reliable regardless of whether any users are in the room — it fires and forgets; clients that are connected receive it in real time. The spec requires "all users see progress" which `broadcastEvent` satisfies for currently-connected users; users who join after the task finishes will not see past events (no event replay in Liveblocks), which is acceptable since the canvas itself shows the result.
- The `ai:action` events are dispatched client-side through the existing collaborative flow utilities (not direct Liveblocks Storage writes from the server). This keeps the canvas mutations on the same code path as user interactions: `onNodesChange`/`onEdgesChange` for add operations (so `@liveblocks/react-flow`'s `toLiveblocksInternalNode` wraps them correctly), and `useMutation` for mutate-in-place operations (move, resize, update data, delete). Bypassing this by writing to Storage directly from the server would require replicating the `LiveObject` wrapping logic, which the spec explicitly prohibits.
- A 200ms pause between node broadcasts and edge broadcasts lets React Flow register the added nodes' IDs before edge connection events arrive. Without this pause, edges referencing nodes that haven't been processed yet by React Flow's reconciler can result in orphan edges that don't connect visually until the next render cycle.

- Toolbar interaction guards: `nodrag nopan` classes + `stopPropagation` on `mousedown`/`pointerdown`/`wheel`/`dblclick`. `nodrag` alone was not enough during Feature 14 either — React Flow 12 sometimes claims the pointer for selection-box drag before the class-based short-circuit fires.
- Hover glow uses local `useState` for `hoveredId` rather than CSS `:hover` because inline styles can't express pseudo-selectors and adding a global class per palette entry would spread the palette across two files. `box-shadow: 0 0 4px 1px <textColor>` keeps the glow tight (1px spread, 4px blur) — the spec called out "tight and controlled, not overly blurred".
- Placeholder color: switched from a hardcoded `rgba(255,255,255,0.35)` to `color: textColor; opacity: 0.45`, so light-text-on-dark palettes look right AND dark-text-on-light palettes (amber, cyan) still get a legible dimmed placeholder. Using `opacity` on the span is fine here because the span has no other children.
- Toolbar placement uses `bottom: calc(100% + 10px)` on the node's `position: relative` root — same coordinate frame the shape and handles use — so it tracks the node during drag and never overlaps the shape. `zIndex: 10` keeps it above the shape/handles without interfering with the node-resizer handles (which are outside this DOM subtree).

## Session Notes (14)
- `<NodeResizer>` is added inside the custom node and only sets `isVisible`/`min*` props; it dispatches `dimensions` + `position` changes through the same `onNodesChange` that `useLiveblocksFlow` already wraps, so resize syncs to Storage without any custom mutation. Verified the `NODE_BASE_CONFIG` in `@liveblocks/react-flow` makes `position` atomic and treats `width`/`height` as deep-synced fields.
- Label writes can't go through `onNodesChange` with a `replace` change because `useLiveblocksFlow` syncs `data` as a (deep) LiveObject — replacing the whole node would re-create the LiveObject and break references for in-flight selection/drag state. Direct LiveObject write via `useMutation` (`storage.get("flow").get("nodes").get(id).get("data").set("label", value)`) is the path that keeps the existing node identity and patches just the label key.
- Existing `data.label` reads come from the controlled `nodes` array returned by `useLiveblocksFlow`, which already subscribes to Storage. Using it as the textarea's `value` keeps the typing experience real-time without local mirror state and lets concurrent updates from other clients show up live.
- Label + textarea share one `position: absolute; inset: 0; display: flex; align-items: center; justify-content: center` container; swapping the inner child in place is what satisfies "no layout shift" — sizes/positions are identical between the two modes.
- Textarea uses `rows={1}` and auto-resizes on input by setting `style.height = "auto"` then `scrollHeight + "px"`. Initial mount runs the same measurement inside the `editing` effect so labels that were already multi-line open at the correct height.
- `className="nodrag nopan nowheel"` plus `stopPropagation` on `mousedown`/`pointerdown`/`wheel` was needed (just `nodrag`/`nopan` was not enough in 12.10.2 — wheel scrolling inside the textarea still bubbled to the pane zoom, and React Flow occasionally claimed the pointer for selection-box drag before nodrag could short-circuit).
- Initial `useEffect` had a second effect that called `setEditing(false)` when the node became deselected. React 19's `react-hooks/set-state-in-effect` flagged it, and it was redundant anyway — clicking off the node loses focus on the textarea, which triggers `onBlur` → `setEditing(false)`. Removed.
- Empty-label rendering: when `data.label` is falsy the span renders `"Add label"` at the same centered position with a dimmer color (`rgba(255,255,255,0.35)`), so the placeholder occupies the same slot a real label would, matching the spec's "same centered position" requirement. The textarea also carries `placeholder="Add label"` so the same hint shows while editing an empty node.

## Session Notes (13)
- Shape rendering split into a shared `ShapeVisual` component so the same renderer powers both the node and the drag preview — keeps shapes visually identical and avoids drift.
- `preserveAspectRatio="none"` + `vectorEffect="non-scaling-stroke"` is what lets diamond/hexagon/cylinder stretch to arbitrary node sizes without distorting the stroke width.
- HTML5 drag image element needs to be in the DOM and rendered (not `display: none`) when `setDragImage` is called, so the off-screen previews use `position: absolute; left: -10000px` rather than `display: none` or `visibility: hidden`.
- Drag image is centered on the cursor by passing `(width/2, height/2)` as the offset to `setDragImage` — matches the drop handler which centers the node on `clientX`/`clientY`.
- Fix carried over from feature 12: `canvas-flow.tsx` previously dropped nodes via a custom `useMutation` that did `storage.get("flow").get("nodes").set(id, node as any)`. That stored a plain object, so as soon as React Flow tried to drag/measure/select the node `@liveblocks/react-flow` crashed with `node.setLocal is not a function` or `node.get is not a function`. Insertion now goes through `onNodesChange([{ type: "add", item: node }])`, which is the library's own path and runs the internal `toLiveblocksInternalNode` wrapper (LiveObject + `NODE_BASE_CONFIG` with atomic `position` and local-only `selected`/`dragging`/`measured`/`resizing`).
- Legacy data heal: rooms persisted by the broken handler still contain plain-object nodes. `canvas-flow.tsx` adds a `removeLegacyNodes` mutation + `useEffect` that runs once on mount — it scans `storage.flow.nodes` for entries missing a LiveObject `.get` method, deletes them, then re-adds them through `onNodesChange` so they get re-wrapped. Subsequent loads are no-ops. The mutation runs synchronously inside the effect, before React Flow's ResizeObserver fires, so the first measurement pass operates on healthy LiveObjects.
- MiniMap shapes: by default `<MiniMap>` renders every node as a rounded rect. `components/editor/canvas/minimap-shape.tsx` is a custom `nodeComponent` that subscribes to each node's data via `useNodesData(id)` and emits the matching SVG primitive: `<rect rx={5}>` for rectangle, `<rect rx={height/2}>` for pill, `<ellipse>` for circle, `<polygon>` for diamond/hexagon, and a body `<path>` + top `<ellipse>` for cylinder. Passed to `<MiniMap nodeComponent={MinimapShape}>` so the minimap visualization matches the canvas.

## Architecture Decisions

- Using shadcn/ui on Tailwind v4: CSS variables defined in globals.css via `@theme inline` (no tailwind.config.js)
- Dark-only theme: CSS variables set on `:root`, no light-mode override
- shadcn components live in `components/ui/` — generated by CLI, not modified after installation
- cn() helper in `lib/utils.ts` uses clsx + tailwind-merge
- Animation utilities (animate-in, fade-in-0, zoom-in-95, slide-in-from-*) defined via `@utility` in globals.css for Tailwind v4 compatibility
- `components.json` present at project root for shadcn CLI configuration
- Editor chrome components live in `components/editor/` — hand-authored, not generated
- Project sidebar is `fixed` positioned (floats above canvas, does not push content); slide-in via CSS transform transition
- Auth uses Clerk v7 (`@clerk/nextjs` ^7.3.0); proxy.ts (not middleware.ts) per Next.js 16 convention
- ClerkProvider is inside `<body>` per Clerk v7 requirement; uses `dark` theme + CSS var overrides
- Clerk appearance uses `shadcn` + `dark` themes from `@clerk/ui/themes`, with `@clerk/ui/themes/shadcn.css` imported in globals.css
- Auth pages use route group `app/(auth)/` with shared two-panel layout; catch-all segments `[[...sign-in]]` and `[[...sign-up]]` for Clerk's multi-step flows
- Protected-first middleware: all routes protected except NEXT_PUBLIC_CLERK_SIGN_IN_URL, NEXT_PUBLIC_CLERK_SIGN_UP_URL, and `/api/projects(.*)`. Project API routes enforce Clerk auth inside route handlers so they can return the Feature 06-required JSON `401`.
- Project metadata is stored in PostgreSQL through Prisma. Canvas JSON is externalized to Vercel Blob at a stable `projects/${projectId}/canvas.json` path (public access, no random suffix, overwrite on save); the resulting URL is stored on `Project.canvasBlobUrl` (`String?`, nullable — a project has no saved canvas until the first autosave writes to Blob).
- Prisma Client singleton lives in `lib/prisma.ts`, uses `accelerateUrl` for `prisma+postgres://` URLs and `@prisma/adapter-pg` for direct Postgres URLs, and is cached on `globalThis` outside production.
- Editor workspace URLs use `/editor/[projectId]`; Feature 07 aligns the project ID and Liveblocks room ID by allowing `POST /api/projects` to accept a sanitized slug-style client-generated ID while preserving Prisma's default ID strategy when no ID is supplied.
- `lib/project-access.ts` is the server-side access layer; `lib/project-data.ts` is the server-side data-fetch layer for the sidebar. The two are intentionally separate so the workspace page can do one access check and one full project-list fetch in parallel.
